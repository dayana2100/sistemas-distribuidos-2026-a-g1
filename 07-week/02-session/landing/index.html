DDD: dividir el problema por negocio antes de dividir el despliegue

Domain-Driven Design ayuda a separar el sistema por contextos de negocio. En esta clase el core se divide en tres dominios: Seguridad, Venta e Inventario. Cada uno tiene sus propias entidades, reglas y casos de uso.

Seguridad

Controla autenticacion, autorizacion, tokens JWT, usuarios y permisos.

Venta

Administra pedidos, pagos, clientes, reglas comerciales y estados de la venta.

Inventario

Gestiona existencias, catalogo, reservas, disponibilidad y movimientos de stock.

Idea clave: primero se separa el lenguaje del negocio; despues se decide si ese dominio vive en un proyecto unico, en modulos o en microservicios independientes.

Ejemplo funcional conceptual: Core XYZ

Un usuario inicia sesion en Seguridad, genera una compra en Venta y finalmente Inventario descuenta unidades. Aunque no se implementa la logica real, la demo representa visualmente ese recorrido.

1. Login y validacion del token
2. Registro del pedido y validacion comercial
3. Reserva o descuento de stock

ADR: documentar por que se tomo una decision de arquitectura

Un ADR (Architecture Decision Record) no describe todo el proyecto; solo captura una decision importante, el contexto, las opciones consideradas y las consecuencias.

Cuando escribir un ADR

  • Cuando se cambia la estructura del codigo.
  • Cuando se decide separar un modulo a otro repo.
  • Cuando se introduce API Gateway, Kubernetes o nuevas politicas por ambiente.

Que debe responder

  • Cual era el problema.
  • Que opciones existian.
  • Cual opcion se eligio.
  • Que ganamos y que costo aceptamos.

ADR sugerido para esta clase

Decision: organizar primero por modulo dentro del core
Motivo: reducir confusion entre Seguridad, Venta e Inventario
Siguiente paso: separar a microservicios cuando cada modulo lo justifique
Impacto: mejora la cohesion y facilita desplegar por dominio

Lectura recomendada

En esta demo se incluye un ADR real de ejemplo en docs/adr/0001-organizacion-del-core.md.

Escenario All Project: una sola base de codigo organizada por capas globales

En este enfoque todo vive en el mismo proyecto. Primero se agrupan las carpetas por responsabilidad tecnica y dentro de cada capa aparecen Seguridad, Venta e Inventario.

Estructura representada en el resumen

Entidad/
  Seguridad/
  Venta/
  Inventario/
Repositorio/
  Seguridad/
  Venta/
  Inventario/
Servicio/
  Seguridad/
  Venta/
  Inventario/
Controlador/
  Seguridad/
  Venta/
  Inventario/
Utils/
  validations/
  jwt/

Ventajas

  • Es simple de arrancar y de explicar al equipo.
  • Todo esta en un solo repo con ramas develop, qa y main.
  • Facilita reutilizar utilidades globales al inicio del proyecto.

Riesgos

  • La cohesion del dominio se diluye porque una regla de negocio cruza muchas carpetas.
  • El acoplamiento tecnico crece cuando todos comparten la misma capa global.
  • Escalar a repos separados cuesta mas porque el dominio no esta encapsulado.
Criterio Lectura del escenario
Cohesion Media. El negocio esta repartido entre muchas carpetas tecnicas.
Acoplamiento Alto. Cambiar Venta puede tocar entidades, servicios y controladores mezclados con otros dominios.
Escenario ideal Equipos pequenos o proyectos que todavia no necesitan separacion por dominio.

Escenario By Module: cada dominio junta sus capas y se vuelve mas entendible

Aqui cambia la prioridad. En vez de pensar primero en entidades o servicios globales, se piensa primero en el modulo de negocio y luego en sus capas internas.

Lectura de arquitectura: la carpeta cuenta una historia mas clara. Si alguien entra al modulo Venta, encuentra casi todo lo que pertenece a ese dominio.

Por que mejora

  • Aumenta la cohesion porque el dominio queda junto.
  • Baja el acoplamiento entre modulos.
  • Hace mas natural convertir un modulo en otro repo.

Que se cuida

  • Las utilidades compartidas deben quedarse en una zona comun minima.
  • No todo helper merece vivir fuera del modulo.
  • Las validaciones globales deben ser pocas y estables.

Escenario 1

Seguridad se puede convertir en un nuevo git sin romper el resto del core.

Escenario 2

Venta evoluciona a otro repo cuando su ciclo de despliegue requiere autonomia.

Escenario 3

Inventario se independiza cuando necesita tecnologia, equipo o versionado propio.

Escenario Que gana el equipo Nuevo costo
Repos separados por dominio Versionado, despliegue y ownership mas claro. Mas coordinacion y mas infraestructura.
Tecnologias distintas Cada dominio usa el stack que le conviene. Se complica la observabilidad y la operacion.
Gateway al frente El front no necesita conocer todos los servicios internos. El gateway se vuelve un punto importante de control.

API Gateway: una sola puerta de entrada para todos los ambientes

El gateway concentra autenticacion externa, normaliza rutas y dirige las peticiones al microservicio correcto del mismo ambiente. No mezcla main con qa ni dev.

Front main debe conversar con gateway main y este con los micros de main. Lo mismo aplica para qa y dev. Cada ambiente es una linea aislada.

Que se busca enseñar

El gateway no reemplaza a DDD. Solo ordena el acceso. La separacion real del negocio sigue estando en Seguridad, Venta e Inventario.

Kubernetes: como llevar estos escenarios a un cluster real

Kubernetes no cambia el dominio; cambia la forma de operar. El mapa mas claro para esta clase es separar por namespaces de ambiente y desplegar dentro de cada namespace el front, gateway y microservicios del mismo entorno.

namespace: distribuidos-dev

front-dev, gateway-dev, micro-1-dev, micro-2-dev, micro-3-dev

namespace: distribuidos-qa

front-qa, gateway-qa, micro-1-qa, micro-2-qa, micro-3-qa

namespace: distribuidos-main

front-main, gateway-main, micro-1-main, micro-2-main, micro-3-main

Recursos minimos

  • Namespace por ambiente.
  • Deployment por servicio.
  • Service ClusterIP para trafico interno.
  • Ingress para exposicion externa.
  • ConfigMap y Secret para configuracion.

Buenas practicas

  • No mezclar trafico entre ambientes.
  • Versionar manifests o Helm charts por repo.
  • Centralizar observabilidad y logs.
  • Escalar microservicios de forma independiente.

Ejemplo de lectura operacional

Usuario -> Ingress -> front-main
Usuario -> Ingress -> api-gateway-main
api-gateway-main -> Service micro-1-main
api-gateway-main -> Service micro-2-main
api-gateway-main -> Service micro-3-main
